Punto de partida

¿Qué es la Programación Orientada a Objetos?

Es un paradigma: una forma de resolver problemas estableciendo un paralelismo con la realidad. En vez de instrucciones sueltas, agrupamos datos y comportamiento en clases.

Reutilizable

Escribo una vez, uso muchas.

Legible

El código se parece al problema.

Mantenible

Un cambio queda contenido en un lugar.

Vocabulario

Clase · Objeto · Instancia · Tipo

Clase

La plantilla. Define qué características tiene un tipo de objeto, sin valores. Ej: la clase Puerta.

Tipo

El tipo de dato que define esa clase.

Objeto / Instancia

Un espacio real en memoria (RAM) con valores concretos. Ej: esta puerta de madera, marrón, de 2m.

Con una clase puedo crear tantos objetos como necesite. Cada uno ocupa su propio lugar en memoria.

Se crea con la palabra reservada new:

const miPuerta: Puerta = new Puerta();
Estructura

Anatomía de una clase

Nombreque la identifica.
Visibilidadnivel de exposición.
Atributoslas variables (el estado).
Métodosel comportamiento.
Constructormétodo especial que se ejecuta al hacer new. Inicializa el objeto y no declara tipo de retorno.

Atributos + métodos = miembros de la clase.

this es la autorreferencia: "el atributo de esta instancia", para distinguirlo de un parámetro homónimo.

TypeScript

Ejemplo: la clase Person

TypeScript ofrece un atajo para inicializar atributos desde el constructor: parameter properties.

class Person {
  constructor(
    private name: string,
    private lastname: string
  ) {}
}

Lo vemos en profundidad más adelante.

Pilar 01

Encapsulamiento

Ocultar el cómo y exponer solo el qué. La clase es una "caja hermética": sus miembros no se tocan desde afuera salvo que lo permitamos.

Convención: todo atributo privado, accedido por getters (leer) y setters (escribir).

  • ¿Solo lectura? → solo getter.
  • ¿Solo escritura? → solo setter.
  • Si fuera público, tendría las dos obligatoriamente.
Beneficio

Puedo cambiar la implementación interna sin romper a quien me usa.

Pilar 02

Abstracción

Quedarme con lo esencial del problema e ignorar el resto. Elijo qué atributos y comportamientos importan en mi contexto.

Modelar una Persona…

…para un banco ≠ modelarla para un hospital. El contexto decide.

La POO nos da clases abstractas e interfaces para abstraer.

Pilar 04

Polimorfismo — mismo llamado, distinto comportamiento

Objetos distintos responden diferente al mismo método (misma firma). Surge de tres mecanismos:

Sobreescritura — dentro de una jerarquía.
Interfaces — naturalezas distintas, trato igual.
Composición — cambiar comportamiento en runtime.
Segunda parte · UML

Un lenguaje visual para modelar software

UML (Unified Modeling Language) nos deja diseñar y comunicar antes de programar. Es como el plano del arquitecto. Nosotros nos concentramos en el diagrama de clases: clases, sus atributos y métodos, y las relaciones entre ellas.

Regla de oro: el UML debe ayudar a entender, no a decorar. Si un diagrama confunde, sobra.

Relación 1/6 · ──▷

Herencia — "es un"

classDiagram
    Person <|-- Employee
    Person <|-- ProviderEmployee
    class Person { #name: string }
    class Employee { +file: number }
    class ProviderEmployee { +companyCode }
class Employee extends Person {}
Cómo la reconozco
  • Pregunta clave: "¿X es un Y?" — "un perro es un animal" ✓, "un auto es un motor" ✕.
  • En el código aparece extends.
  • La subclase puede agregar y sobreescribir miembros.
Más ejemplos: Animal ◁ Perro/Gato · Figura ◁ Circulo · Exception ◁ EmployeeException.
⚠️ Error típico: heredar solo para reutilizar código sin un "es un" real. Si no lo hay, preferí composición.
Relación 2/6 · ┈┈▷

Realización — "se comporta como"

classDiagram
    class Turnable {
        <<interface>>
        +turnOn() boolean
        +turnOff() boolean
    }
    Turnable <|.. Engine
    Turnable <|.. AirConditioner
class Engine implements Turnable {}
Cómo la reconozco
  • Pregunta: "¿X se comporta como Y?" sin ser lo mismo.
  • Del lado de Y aparece <<interface>>; en código, implements (puede ser varias).
  • La interfaz no tiene código, solo firmas → obliga a implementarlas.
Más ejemplos: Comparable ◁┈ Producto · Serializable ◁┈ Usuario · Volador ◁┈ Pajaro/Avion.
💡 Herencia comparte qué es; la interfaz comparte qué sabe hacer.
Relación 3/6 · ───▶

Asociación — "usa / conoce a"

classDiagram
    Profesor --> Curso : dicta

La relación más genérica: un objeto conoce a otro y colabora, pero cada uno tiene vida propia. Cuando dudes, "asociación" es el default.

Cómo la reconozco
  • Pregunta: "¿X usa / conoce a Y de forma permanente?"
  • X guarda a Y como atributo, pero no lo posee como parte suya.
  • Suele llevar verbo ("dicta", "atiende") y multiplicidad.
Ejemplos: Medico → Paciente · Cliente → Pedido · Alumno "*"→"*" Materia.
Agregación y composición son tipos especiales de asociación (un "tiene un" más fuerte).

◇── débil

◆── fuerte
Composición

El todo no existe sin la parte.

Auto ◆— Motor.

classDiagram
    Vehicle o-- AirConditioner : débil

Cómo las distingo

Pregunta: "si destruyo el todo, ¿la parte sigue teniendo vida propia?"

Sí sobrevive → agregación (rombo vacío). La parte se recibe desde afuera.
No sobrevive → composición (rombo lleno). El todo la crea y la destruye.
🧭 Rombo lleno = "vida y muerte juntas"; rombo vacío = "vidas independientes". El rombo va del lado del todo.
// composición: el todo CREA la parte
habitaciones = [new Habitacion()];
// agregación: la RECIBE de afuera
constructor(private jugadores: Jugador[]) {}
Relación 6/6 · ┈┈▶

Dependencia — "usa de paso"

classDiagram
    PersonManager ..> Employee : parámetro
class PersonManager {
  getName(employee: Employee): string {
    return employee.getName();
  }
}
Cómo la reconozco
  • Pregunta: "¿X usa a Y solo por un rato y se olvida?"
  • Y es parámetro, variable local o retorno — nunca atributo guardado.
De más débil a más fuerte
Dependencia → Asociación → Agregación → Composición

En código

Clase abstracta vs. Interfaz

Clase abstracta — código + obligación
abstract class Vehicle {
  constructor(protected engine: Engine) {}

  honk(): void { log("¡Tu-tuuuu!"); }  // concreto

  abstract turnEngineOn(): boolean;  // abstracto
}

class Car extends Vehicle {
  turnEngineOn(): boolean {   // OBLIGATORIO
    return this.engine.turnOn();
  }
}
Interfaz — solo el contrato
interface Turnable {
  turnOn(): boolean;
  turnOff(): boolean;
}

class Engine implements Turnable {
  turnOn(): boolean {
    log("Encendiendo el motor");
    return true;
  }
  turnOff(): boolean {
    log("Apagando el motor");
    return true;
  }
}
Criterio de decisión

¿Cuándo uso cada una?

Clase abstracta cuando…
  • Hay una relación "es un" clara (jerarquía).
  • Querés compartir código entre subclases.
  • Necesitás atributos con estado común.
Interfaz cuando…
  • Definís un contrato de comportamiento.
  • Objetos de naturaleza distinta se tratan igual.
  • Una clase debe cumplir varios contratos.
💡 Buena práctica: programar contra interfaces, no contra implementaciones concretas → código intercambiable y fácil de mantener.
