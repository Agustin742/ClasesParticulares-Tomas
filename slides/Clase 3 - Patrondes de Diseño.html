Clase 03 · Patrones de diseño
Definición

¿Qué es un
patrón de diseño?

Una solución general y reutilizable a un problema de diseño que se repite.

No es código para copiar y pegar: es una plantilla — una forma de organizar clases e interfaces que vos adaptás a tu caso.

Estudiaron cómo los buenos programadores resolvían los mismos problemas… y les pusieron nombre.

El origen
Gang of Four
Gamma · Helm · Johnson · Vlissides
Design Patterns — 1994
Analogía · ajedrez

«Enroque» o «gambito de dama» son patrones. No te dan la partida entera: te dan una jugada conocida para una situación conocida.

Si sabés el nombre, tu compañero te entiende en una palabra.

Creacionales

Cómo se crean los objetos.

El problema típico

Tu código hace new ClaseConcreta() por todos lados y queda casado con esa clase. El día que necesitás otra, tenés que tocar diez lugares.

Estos patrones encapsulan la creación detrás de una abstracción: el resto del código pide «dame un X» sin saber cuál concreto le toca.

Analogía · restaurante

Pedís «una pizza»; no entrás a la cocina a armarla. La cocina (la fábrica) decide cómo se construye y con qué.

Qué comparten

Separan qué se usa de cómo se construye.

Estructurales

Cómo se arman y relacionan los objetos.

El problema típico

Tenés piezas que no encajan, o querés sumar / esconder / organizar comportamiento sin reescribir lo que ya funciona.

Armás estructuras más grandes sin que todo quede rígido: cambiás envolviendo o reacomodando, no rompiendo.

Qué comparten

Se apoyan en composición (un objeto contiene a otro) para reorganizar sin depender de la herencia.

Comportamiento

Cómo se comunican los objetos.

El problema típico

La lógica de «quién hace qué y cuándo» termina enredada en if gigantes, o en objetos que se conocen demasiado entre sí.

Resuelven quién le habla a quién, quién decide qué y cómo fluye una petición.

Analogía · oficina

No importa quién es cada empleado, sino el protocolo: cómo circula un trámite, a quién se le avisa, quién delega en quién.

Qué comparten

Desacoplan al que pide del que hace, metiendo una interfaz en el medio. Es la familia más numerosa (10).

Creacional

«No sé qué clase concreta instanciar / quiero controlar la creación / el new está desparramado.»

Palabras que lo delatan

crear · instanciar · new · única instancia · familia de productos · construir paso a paso

Estructural

«Hacer encajar cosas incompatibles / envolver / simplificar una estructura / representar un todo-y-sus-partes.»

Palabras que lo delatan

adaptar · envolver · compatibilizar · árbol · jerarquía · esconder complejidad · agregar sin tocar

Comportamiento

«Elegir un comportamiento / avisarle a varios cuando algo pasa / pasar una petición por una cadena.»

Palabras que lo delatan

avisar · notificar · suscribir · intercambiar algoritmo · en tiempo de ejecución · cadena · pasos fijos

Intención

Proporcionar una interfaz para crear objetos, dejando que las subclases decidan qué clase concreta instanciar.

El código cliente trabaja contra la abstracción del producto, no contra las clases concretas.

En criollo

En vez de new ProductoConcreto() desparramado, centralizás la creación en un método fábrica que cada subclase implementa a su manera.

Factory Method · señal delatora

Sospechalo
cuando…

  • Tenés new ClaseConcreta() repetido en muchos lugares, y la clase concreta va a cambiar o crecer.
  • Un if/switch decide qué objeto crear según un tipo. 👈 el mismo olor de OCP, pero en la creación.
  • Querés que quien usa el objeto no dependa de cuál concreto es.
transporte.ts · el producto
// Producto: la abstracción con la que trabaja el cliente interface Transporte { entregar(): string; } class Camion implements Transporte { entregar(): string { return "Entrega por tierra en caja"; } } class Barco implements Transporte { entregar(): string { return "Entrega por mar en contenedor"; } }
🤔 Pará y pensá — el if que crece Antes de ver la solución
imaginá resolverlo con un switch
function crearTransporte(tipo: string): Transporte { if (tipo === "camion") return new Camion(); if (tipo === "barco") return new Barco(); // ... }
  • En un sistema real ese mismo tipo se repite en otros switch (uno para el costo, otro para el ícono, otro para el seguimiento…).
  • Si agregás Avion, ¿en cuántos lugares te tenés que acordar de sumar el caso? ¿Qué pasa si te olvidás de uno?
  • Pista de la Clase 2: esto es el olor exacto de OCP. ¿Con qué lo curábamos? 🧩
logistica.ts · el creador
abstract class Logistica { // las subclases deciden QUÉ transporte crear protected abstract crearTransporte(): Transporte; // lógica que NO cambia y no conoce la clase concreta planificarEntrega(): string { const t = this.crearTransporte(); // ← método fábrica return `Planificando: ${t.entregar()}`; } } class LogisticaTerrestre extends Logistica { protected crearTransporte(): Transporte { return new Camion(); } } class LogisticaMaritima extends Logistica { protected crearTransporte(): Transporte { return new Barco(); } } const l: Logistica = new LogisticaMaritima(); console.log(l.planificarEntrega()); // Planificando: Entrega por mar en contenedor
El creador no sabe

La lógica que usa el transporte vive en el creador y no sabe cuál concreto le toca. Cada subclase decide qué producto fabricar.

¿Aparece un Avion? Nueva subclase LogisticaAerea, y no tocás planificarEntrega().

Patrón 02 / 05 · paso: intención
Adapter
Molde: intención → señal → código → UML → SOLID
Intención

Hacer que dos interfaces incompatibles trabajen juntas.

El Adapter envuelve una clase existente (que no podés o no querés modificar) y le pone por delante la interfaz que tu código espera.

Analogía · el adaptador de enchufe

El tomacorriente de la pared (interfaz A) y tu cargador (interfaz B) no encajan. No cambiás ni la pared ni el cargador: metés un adaptador en el medio.

Adapter · señal delatora

Sospechalo
cuando…

  • Querés usar una librería / clase vieja / API externa cuyos métodos no encajan con lo que el resto de tu código espera.
  • No podés (o no conviene) tocar esa clase — código de terceros, sistema legacy.
  • Ves que traducís «a mano» de un formato a otro en muchos lugares.
clima.ts · lo mío vs. lo ajeno
// Lo que MI código espera usar en todos lados: interface ProveedorClima { temperatura(ciudad: string): number; } // Librería externa, con OTRA interfaz (no la puedo tocar): class WeatherApiExterna { fetchWeatherFahrenheit(city: string): number { return 68; // supongamos que devuelve °F } }
clima-adapter.ts · implementa + compone
class ClimaAdapter implements ProveedorClima { private api = new WeatherApiExterna(); // ← composición: envuelve al "adaptee" temperatura(ciudad: string): number { const f = this.api.fetchWeatherFahrenheit(ciudad); return Math.round((f - 32) * 5 / 9); // traduce °F → °C } } // El cliente habla SOLO con ProveedorClima; ni se entera de la API externa: function mostrarClima(prov: ProveedorClima) { console.log(`${prov.temperatura("Rosario")} °C`); } mostrarClima(new ClimaAdapter()); // 20 °C
La jugada

El adaptador implementa la interfaz que espero y, por dentro, se compone de la clase incompatible — traduciendo la llamada.

Cambiar de proveedor = otro adapter. El cliente ni se entera.

Patrón 03 / 05 · paso: intención
Decorator
Molde: intención → señal → código → UML → SOLID
Intención

Agregar responsabilidades a un objeto de forma dinámica, envolviéndolo, sin tocar su clase y sin usar herencia.

La clave: el decorador tiene la misma interfaz que el objeto que envuelve — quien lo usa no nota la diferencia… pero por dentro le suma comportamiento.

Analogía · el café ☕

Empezás con un espresso. Le agregás leche (lo envolvés), le agregás caramelo (lo volvés a envolver). Sigue siendo «una bebida que cuesta $X y se describe», pero cada capa suma precio y descripción.

🤔 Razoná — ¿por qué no con herencia? Contá las clases

Un café puede llevar leche, caramelo y/o crema, en cualquier combinación. Si lo resolvieras con una subclase por combinación (CafeConLeche, CafeConLecheYCaramelo…):

  • Contá cuántas clases necesitás para cubrir esas 3 opciones en toda combinación. ¿Y si mañana entra una cuarta?
  • ¿Qué hacés si querés decidir los agregados en runtime, según lo que pide el cliente en el momento?

Decorator · señal delatora

Sospechalo
cuando…

  • Querés combinar funcionalidades opcionales en cualquier orden, y por herencia te explotarían las subclases (💥 combinatoria).
  • Necesitás agregar comportamiento en tiempo de ejecución, no de compilación.
  • No querés (o no podés) modificar la clase original.
cafe.ts · la interfaz y el concreto base
interface Cafe { costo(): number; descripcion(): string; } class Espresso implements Cafe { costo(): number { return 1500; } descripcion(): string { return "espresso"; } }
cafe-decorator.ts · implementa Cafe + contiene un Cafe
// Decorador base: mismo tipo que el envuelto, y lo compone abstract class CafeDecorator implements Cafe { constructor(protected base: Cafe) {} abstract costo(): number; abstract descripcion(): string; } class ConLeche extends CafeDecorator { costo(): number { return this.base.costo() + 300; } descripcion(): string { return this.base.descripcion() + " + leche"; } } class ConCaramelo extends CafeDecorator { costo(): number { return this.base.costo() + 500; } descripcion(): string { return this.base.descripcion() + " + caramelo"; } } // Envuelvo en capas, en el orden que quiera: let bebida: Cafe = new ConCaramelo(new ConLeche(new Espresso())); console.log(bebida.descripcion(), bebida.costo()); // espresso + leche + caramelo 2300
La jugada

El decorador implementa Cafe y guarda un Cafe adentro (composición). Delega en el envuelto y le suma lo suyo.

Apilás capas en el orden que quieras — y en runtime.

Patrón 04 / 05 · paso: intención
Strategy
Molde: intención → señal → código → UML → SOLID
Intención

Definir una familia de algoritmos, encapsular cada uno en su clase y hacerlos intercambiables.

El objeto que los usa (el contexto) puede cambiar de algoritmo —incluso en tiempo de ejecución— sin cambiar su propio código.

El material lo llama…

«El patrón más representativo del polimorfismo»: composición + interfaz en su forma más pura. Si entendiste el ejemplo de Pago de la Clase 2, ya entendiste Strategy.

Strategy · señal delatora

Sospechalo
cuando…

  • Tenés un if/switch que elige cómo hacer algo (cómo ordenar, calcular el envío, comprimir) y esos «cómo» van a crecer.
  • Querés cambiar el algoritmo en runtime (el usuario elige método de pago, tipo de envío…).
  • Varias clases hacen lo mismo con pequeñas variantes y querés no duplicar.
envio.ts · la familia de algoritmos
interface EstrategiaEnvio { costo(pesoKg: number): number; } class EnvioEstandar implements EstrategiaEnvio { costo(pesoKg: number): number { return pesoKg * 100; } } class EnvioExpress implements EstrategiaEnvio { costo(pesoKg: number): number { return pesoKg * 250 + 500; } }
🤔 Predecí — cambiar sin tocar al cliente Antes de la próxima slide

Tenés EnvioEstandar y EnvioExpress. El Carrito tiene que poder calcular el total con uno u otro, e incluso cambiar de tipo de envío después de creado.

¿Qué tendría que guardar Carrito para no quedar atado a un envío concreto?

La pista de siempre

No una clase concreta, sino una referencia a la interfaz EstrategiaEnvio.

Composición + interfaz. El contexto delega y no sabe cuál estrategia le toca.

carrito.ts · el contexto compone y delega
class Carrito { constructor(private envio: EstrategiaEnvio) {} setEnvio(e: EstrategiaEnvio) { this.envio = e; } // ← cambio en runtime total(pesoKg: number, subtotal: number): number { return subtotal + this.envio.costo(pesoKg); // delega, no sabe cuál es } } const c = new Carrito(new EnvioEstandar()); console.log(c.total(2, 5000)); // 5200 c.setEnvio(new EnvioExpress()); // cambio la estrategia sin tocar Carrito console.log(c.total(2, 5000)); // 6000
La jugada

El contexto guarda una estrategia (composición contra la interfaz) y delega en ella. Puede recibirla o cambiarla cuando quiera.

Agregar EnvioInternacional = una clase nueva. Carrito ni se entera.

Patrón 05 / 05 · paso: intención
Observer
Molde: intención → señal → código → UML → SOLID
Intención

Definir una dependencia uno-a-muchos: cuando uno (el sujeto/observado) cambia de estado, todos los que dependen de él (los observadores) se enteran y reaccionan automáticamente.

Analogía · el canal de YouTube ▶

El canal (sujeto) sube un video; todos los suscriptores (observadores) reciben la notificación, sin que el canal sepa quién es cada uno. Te suscribís (attach) y te desuscribís (detach) cuando querés.

Observer · señal delatora

Sospechalo
cuando…

  • Un cambio en un objeto tiene que disparar reacciones en varios otros, y no querés que el primero conozca a cada uno.
  • Escuchás palabras como notificar, avisar, suscribir, «cuando pase X, que se actualicen Y y Z», eventos.
  • El conjunto de «interesados» cambia con el tiempo (se suman, se van).
observer.ts · los dos contratos
interface Observer { update(estado: string): void; } interface Subject { attach(o: Observer): void; detach(o: Observer): void; notify(): void; }
🤔 Pará y pensá — el costo de conocer a todos Avisar por nombre, hardcodeado
avisar por nombre… 😬
subirVideo(t: string) { ana.avisar(t); beto.avisar(t); // ... }
  • Cuando se suscribe Caro, ¿qué tenés que tocar? ¿Y si Ana se va?
  • ¿Canal debería conocer a cada suscriptor concreto para hacer su trabajo?

Cada alta o baja te obliga a editar Canal. La cura: que Canal hable con una lista de una interfaz (Observer) y no sepa quién hay del otro lado. Eso es Observer.

canal.ts · el sujeto guarda la lista y notifica
class Canal implements Subject { private observers: Observer[] = []; private ultimoVideo = ""; attach(o: Observer) { this.observers.push(o); } detach(o: Observer) { this.observers = this.observers.filter(x => x !== o); } notify() { this.observers.forEach(o => o.update(this.ultimoVideo)); } subirVideo(titulo: string) { this.ultimoVideo = titulo; this.notify(); // ← avisa a TODOS los suscriptos } } class Suscriptor implements Observer { constructor(private nombre: string) {} update(video: string) { console.log(`${this.nombre} → nuevo video: ${video}`); } } const canal = new Canal(); canal.attach(new Suscriptor("Ana")); canal.attach(new Suscriptor("Beto")); canal.subirVideo("Patrones de diseño en 10 min"); // Ana → nuevo video: ... | Beto → nuevo video: ...
La jugada

El sujeto guarda la lista de observadores y, al cambiar, llama a update() de cada uno.

🛠 Ejercicio en vivo — ¿qué patrón conviene? Decidí + justificá en una frase
A

En un juego online, cuando un jugador sube de nivel tienen que reaccionar el marcador, el chat y el sonido. Van a sumar más reacciones (misiones, ranking). El jugador no debería tener que conocer a ninguna de esas partes.

B

Un editor abre archivos de distinto tipo (texto, planilla, presentación). El procedimiento de «abrir» es siempre el mismo, pero el objeto documento que termina creándose depende de la extensión del archivo.

C

Tu sistema de cobros usa en todos lados Pago.pagar(monto). Integrás MercadoPago, cuyo SDK ofrece createPayment({ amount, currency }), con otros nombres y otro formato. El resto tiene que seguir cobrando igual que siempre.

🛠 Ejercicio en vivo — soluciones Primero la familia, después el patrón
AObserver

Es comunicar: un objeto cambia de estado y varios interesados —que van a crecer— reaccionan sin que el emisor los conozca.

Jugador = Subject; marcador, chat y sonido = Observer. Sumar reacción = un observador nuevo.

BFactory Method

Es crear: el flujo de «abrir» no cambia, pero qué objeto se instancia sí.

Base con abrir() + método fábrica abstracto crearDocumento(); cada subclase decide el tipo concreto.

CAdapter

Es estructura: dos interfaces que no encajan y una (el SDK) que no controlás.

MercadoPagoAdapter implements Pago compone el SDK y traduce pagar(monto) → createPayment(...).

Seguimos con patrones: más del catálogo y cuándo conviene cada uno, con ejemplos y diagramas.

  • Qué es un patrón: una solución probada con nombre a un problema recurrente (GoF, 1994).
  • El mapa de 3 familias y el método: ¿crear, estructurar o comunicar?
  • 5 patrones ancla — Factory Method, Adapter, Decorator, Strategy, Observer — cada uno por su señal delatora.
  • Todos se apoyan en lo mismo: interfaces + composición → SOLID en acción.
📊 ¿Cuál elijo? — los 5 de hoy La pregunta que separa a cada uno
PatrónLa pregunta que lo delataEn una frase
Factory Method¿El problema es qué objeto crear sin atarme al concreto?Subclases deciden la clase concreta.
Adapter¿Tengo dos interfaces que no encajan y una no la controlo?Traduzco una interfaz a la que espero.
Decorator¿Quiero sumar comportamientos combinables sin tocar la clase?Envuelvo manteniendo la misma interfaz.
Strategy¿Quiero elegir cómo hago algo entre varias formas, en runtime?Algoritmo intercambiable por composición.
Observer¿Un cambio debe avisar a varios que aparecen y desaparecen?Uno-a-muchos: el sujeto notifica.
Adapter vs Decorator

Los dos envuelven. Adapter cambia la interfaz (para que encaje); Decorator la mantiene (para sumar comportamiento).

Strategy vs Factory Method

UML casi igual. Factory Method decide qué crear; Strategy decide cómo hacer algo con algo que ya tenés.

✍ Ejercicios — ¿qué patrón aplica? Para cada uno
1 · ¿Qué patrón de los 5? 2 · Justificá: ¿crear, estructurar o comunicar? ¿Por qué ese y no el parecido? 3 · Bosquejá UML + esqueleto en TypeScript.
1

Una app de préstamos calcula intereses con varias fórmulas (tasa fija, variable, promo). Agregan fórmulas seguido y un mismo préstamo puede pasar de una a otra sin recrearse. El objeto préstamo no debería reescribirse cada vez.

2

Un tablero de trading muestra, por acción, un gráfico, un registro de alertas y el total de cartera. Cuando el precio se actualiza, las tres vistas quedan al día. El usuario agrega o quita vistas mientras corre.

3

Tenés Mensaje con contenido() que ya funciona (no tocar). Un mensaje puede ir cifrado, comprimido, firmado o varias a la vez, y quien lo arma decide la combinación en el momento.

4

Un framework de UI define el procedimiento para mostrar una ventana modal (crearla, posicionarla, animar). Es idéntico para todas, pero cada pantalla —login, alerta, formulario— crea su propio contenido interno.

5

Tu módulo de reportes consume FuenteDatos.obtener(): Registro[]. Entra un cliente que expone su info con fetchRecords() en otro formato y otros nombres. Hay que alimentar ese origen sin reescribir el flujo.

6

Una app de navegación ya tiene mapa y destino. El usuario pide la ruta más rápida, la más corta o una que evite peajes, y puede cambiar de criterio sobre la misma búsqueda sin arrancar de cero.

7

En un videojuego, cada nivel genera oleadas. El procedimiento es siempre el mismo (posición, aparecer, activar IA), pero el tipo de enemigo depende del nivel (bosque→lobos, cueva→murciélagos, castillo→caballeros). Sumar un nivel no debería tocar la lógica de oleadas.

Las soluciones, en las próximas dos slides →

✍ Soluciones — 1 a 4 Primero la familia, después el patrón
1Strategy

Comportamiento: no es crear ni estructurar, es elegir cómo se calcula y poder cambiarlo en caliente. CalculoInteres.calcular(capital, plazo) con TasaFija / TasaVariable / TasaPromo; Prestamo compone y delega. Fórmula nueva = clase nueva.

2Observer

Comunicar: un cambio (el precio) dispara reacciones en varias vistas que aparecen y desaparecen. Accion es Subject (attach/detach/notify); gráfico, alertas y cartera son Observer con update(precio).

3Decorator

Estructura: sumar responsabilidades combinables sin tocar Mensaje ni explotar en subclases. MensajeDecorator implements Mensaje y contiene un Mensaje; Cifrado / Comprimido / Firmado envuelven → new Firmado(new Cifrado(base)).

No es Adapter: la interfaz se mantiene, no se traduce.

4Factory Method

Crear: el procedimiento de mostrar la ventana no cambia; cambia qué contenido se crea. Clase base Ventana con mostrar() (flujo fijo) que llama al método fábrica abstracto crearContenido(); VentanaLogin / VentanaAlerta / VentanaForm lo redefinen.

✍ Soluciones — 5 a 7 Ojo con los pares que se parecen
5Adapter

Estructura: una interfaz que no encaja y que no controlás. ServicioAdapter implements FuenteDatos compone el servicio y traduce obtener() → fetchRecords() + mapea el formato.

El flujo de reportes queda intacto.

6Strategy

Los datos ya están (mapa + destino); lo que cambia es cómo se calcula, con cambio de criterio en runtime. CalculadorRuta con MasRapida / MasCorta / EvitaPeajes.

No es Factory Method: no fabricás productos distintos, variás el cálculo sobre datos que ya tenés.

7Factory Method

Crear: la lógica de la oleada no cambia; cambia qué enemigo se instancia. Nivel.lanzarOleada() (flujo fijo) llama a crearEnemigo() abstracto; NivelBosque / Cueva / Castillo devuelven Lobo / Murcielago / Caballero.

Siempre en dos pasos: ¿crear, estructurar o comunicar? → familia. Después, la señal que lo distingue del parecido.
