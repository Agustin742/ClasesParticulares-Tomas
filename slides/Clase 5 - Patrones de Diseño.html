  • Singleton — “una única instancia” accesible desde todo el sistema.
  • Factory Method — un new/switch que decide qué clase concreta crear (un producto).
  • Abstract Factory — una familia de productos que varían juntos y no se mezclan.
  • Builder — objeto con muchas partes opcionales, armado paso a paso.
  • Adapter — dos interfaces que no encajan y una no la controlás.
  • Composite — árbol todo-parte, tratar igual a la hoja y a la rama.
  • Decorator — sumar comportamientos combinables sin tocar la clase.
  • Strategy — elegir cómo hacer algo entre algoritmos, cambiable en runtime.
  • Observer — un cambio tiene que avisarle a varios que se suscriben y se van.

Estructural
Patrón 01 / 04 · intención
Facade
El más simple de los 13: solo simplifica el acceso.
Intención

Ofrecer una interfaz simple y de alto nivel a un subsistema complejo, escondiendo sus partes detrás de una sola puerta de entrada.

No agrega funcionalidad, no traduce nada: solo simplifica el acceso.

Analogía

El mozo del restaurante. Pedís “milanesa con papas” y no hablás con el parrillero, el de las papas, la caja ni el que lava. El mozo (fachada) articula con todos. La cocina sigue siendo compleja: vos no la tocás.

Facade — señal delatora

sospechalo cuando…
  • El cliente tiene que orquestar muchas clases en un orden específico para hacer algo simple, y ese código de orquestación se repite.
  • Querés desacoplar tu código de una biblioteca o subsistema grande del que usás el 10%.
  • El enunciado dice: “esconder la complejidad”, “un punto de entrada único”, “que el cliente no conozca los detalles internos”, “simplificar el uso”.
La fachada
orquesta
— no —
implementa

Si empieza a hacer todo, se vuelve un “objeto dios”. Ojo con SRP.

Facade — el código

el cliente conoce UNA clase, no cuatro
// --- El subsistema: varias clases, cada una // con su propia complejidad class Codec { cargar(a: string) { return `bytes(${a})`; } } class ExtractorAudio { extraer(b: string) { return `audio[${b}]`; } } class Compresor { comprimir(x, fmt: string) { return `${x}→${fmt}`; } } class MarcaDeAgua { aplicar(x: string) { return `${x}+marca`; } } // El cliente: una línea, cero conocimiento // del subsistema const resultado = new ConversorDeVideo().convertir("clase.avi", "mp4");
// --- La FACHADA: una sola puerta, un solo // método de alto nivel class ConversorDeVideo { private codec = new Codec(); private audio = new ExtractorAudio(); private compresor = new Compresor(); private marca = new MarcaDeAgua(); convertir(archivo: string, formato: string): string { const bytes = this.codec.cargar(archivo); // el orden const pista = this.audio.extraer(bytes); // de los pasos, const comp = this.compresor.comprimir(pista, formato); // que antes return this.marca.aplicar(comp); // sabía el cliente, } // ahora vive acá }

Sin la fachada, cada cliente que quiera convertir un video tiene que conocer las 4 clases, sus métodos y el orden correcto. Con la fachada, conoce una.

Razoná — Facade vs Adapter

los dos “envuelven”

Adapter (Clase 3) y Facade envuelven otra cosa y le ponen una interfaz por delante. Desempatalos vos:

  • En Adapter, ¿la interfaz que ofrecés la elegís vos o te la impone el código que ya existe?
  • En Facade, ¿existía antes una interfaz que respetar, o la inventás vos para que sea cómoda?
  • Adapter envuelve una clase; Facade, muchas. ¿Por qué esa diferencia es una consecuencia, no la definición?
Click para revelar

Adapter existe porque hay una incompatibilidad: el cliente ya habla Pago.pagar() y el SDK habla createPayment({...}). La interfaz destino ya estaba dada; tu trabajo es traducir.

Facade existe porque hay complejidad: nadie te obliga a nada, inventás una interfaz cómoda que no existía. Tu trabajo es simplificar.

Adapter traduce porque no encaja. Facade simplifica porque es engorroso. (Y Decorator ni traduce ni simplifica: mantiene la interfaz y suma.)

Intención

Pasar una petición a lo largo de una cadena de manejadores. Cada uno decide si la procesa o si la pasa al siguiente.

El emisor no sabe quién va a atenderla.

Analogía

Un trámite en la facultad. Lo presentás en la ventanilla; si no le corresponde, lo pasa al jefe de departamento, y si tampoco, al decano. Vos entregaste un solo papel y no sabés (ni te importa) quién lo resolvió.

Predecí — el if anidado que crece

un sistema de aprobación de compras
interface Solicitud { monto: number; autenticado: boolean; bloqueado: boolean; } function aprobar(s: Solicitud): string { if (!s.autenticado) return "Rechazada: no autenticado"; else { if (s.monto <= 0) return "Rechazada: monto inválido"; else { if (s.monto > 100000) return "Rechazada: supera el límite"; else { if (s.bloqueado) return "Rechazada: bloqueado"; // ...y seguimos sumando niveles } } } return "Aprobada"; }
  • Córdoba necesita el mismo control pero sin el del límite; Rosario quiere uno nuevo al principio. ¿Copiás la función?
  • ¿Y si te piden cambiar el orden de los controles?
  • Fijate: cada if hace lo mismo — evalúa una condición y, si pasa, le cede el turno al siguiente. ¿Y si cada control fuera un objeto que conoce al que le sigue?

Chain of Responsibility — señal delatora

sospechalo cuando…
  • Una petición debe pasar por una secuencia de pasos/controles/filtros, y cualquiera puede cortar el flujo.
  • El orden de esos pasos cambia o se configura, y se agregan y sacan pasos seguido.
  • El emisor no debe saber quién va a atender la petición.

Chain of Responsibility — el código

el orden es un dato, no está cableado
interface Solicitud { monto: number; autenticado: boolean; } abstract class Handler { private siguiente: Handler | null = null; // devuelve el handler recibido → permite // encadenar: a.encadenar(b).encadenar(c) encadenar(h: Handler): Handler { this.siguiente = h; return h; } // si no corté, le paso la solicitud al que // sigue. Sin siguiente = pasó todos. manejar(s: Solicitud): string { return this.siguiente ? this.siguiente.manejar(s) : "Aprobada"; } }
class ValidarAutenticacion extends Handler { manejar(s: Solicitud): string { if (!s.autenticado) return "Rechazada: no auth"; // ← CORTA return super.manejar(s); // ← SIGUE } } class ValidarMonto extends Handler { manejar(s: Solicitud): string { if (s.monto <= 0) return "Rechazada: monto"; return super.manejar(s); } } class ValidarLimite extends Handler { manejar(s: Solicitud): string { if (s.monto > 100000) return "Rechazada: límite"; return super.manejar(s); } } // Armo la cadena. El ORDEN es un dato: const cadena = new ValidarAutenticacion(); cadena.encadenar(new ValidarMonto()) .encadenar(new ValidarLimite());

La sucursal de Córdoba arma otra cadena con los mismos objetos en otro orden. Cero código nuevo.

Comportamiento
Patrón 03 / 04 · intención
Template
Method
El esqueleto es fijo; cambian los rellenos.
Intención

Definir el esqueleto de un algoritmo en una clase base, dejando que las subclases redefinan algunos pasos sin cambiar la estructura general.

Cátedra: “un proceso siempre requiere los mismos pasos, pero admitiendo variaciones en la implementación, delegables a subclases”.

Analogía

Una receta base de torta. Batir, mezclar, hornear 40 min, desmoldar. Los pasos y el orden no cambian. Lo que cambia es con qué la rellenás: chocolate, vainilla, limón. Nadie hornea antes de mezclar.

Razoná — ¿qué se repite y qué varía?

importar alumnos: CSV vs JSON
Los dos importadores hacen esto
  1. leer el archivo
  2. parsear el contenido
  3. validar que haya datos
  4. guardar en la base
  5. devolver “importados N”
  • ¿Cuáles pasos son idénticos y cuáles cambian?
  • Si escribís dos clases separadas, ¿cuánto duplicás?
  • Si el orden hay que proteger — que nadie guarde sin validar — ¿dónde lo ponés para que una subclase no lo rompa?
Click para revelar

El orden vive en la clase base, en un método que no se redefine (el template method).

Solo los agujeros (leer, parsear) quedan abstractos para que la subclase los rellene.

Template Method — señal delatora

sospechalo cuando…
  • Hay un procedimiento fijo con pasos que varían según el caso: importar datos, generar un reporte, procesar un pago, correr un test.
  • Varias clases hacen casi lo mismo, y la diferencia está en 2 o 3 pasos en el medio.
  • El enunciado dice: “el procedimiento es siempre el mismo”, “los mismos pasos”, “la estructura no cambia, cambia la implementación de algunos pasos”.
La herramienta
Herencia

Los pasos se redefinen; el esqueleto no. Es la gran diferencia con Strategy, que usa composición.

Template Method — el código

el orden está blindado en la clase base
abstract class ImportadorDeDatos { // EL TEMPLATE METHOD: acá vive el ORDEN. // No es abstracto y NO se redefine. importar(ruta: string): string { const crudo = this.leer(ruta); // VARIABLE const datos = this.parsear(crudo); // VARIABLE if (!this.validar(datos)) return "Inválidos"; // hook this.persistir(datos); // FIJO return `Importados ${datos.length}`; } // los agujeros que la subclase DEBE rellenar protected abstract leer(ruta: string): string; protected abstract parsear(c: string): string[]; // "hook": default, la subclase PUEDE cambiarlo protected validar(d: string[]): boolean { return d.length > 0; } // paso común: se escribe UNA sola vez, acá private persistir(d: string[]): void { console.log(`Guardando ${d.length} filas...`); } }
class ImportadorCSV extends ImportadorDeDatos { protected leer(ruta: string): string { return "nombre,edad\nTomas,22\nAna,25"; } protected parsear(c: string): string[] { return c.split("\n").slice(1); // salteo encabezado } } class ImportadorJSON extends ImportadorDeDatos { protected leer(ruta: string): string { return '[{"nombre":"Tomas"},{"nombre":"Ana"}]'; } protected parsear(c: string): string[] { return JSON.parse(c).map((o: any) => o.nombre); } protected validar(d: string[]): boolean { return d.length > 1; // este hook lo pisa } } console.log(new ImportadorCSV().importar("a.csv")); // Importados 2 console.log(new ImportadorJSON().importar("a.json")); // Importados 2

importar() es la única puerta. Ninguna subclase puede persistir sin validar, ni parsear antes de leer: el orden está blindado en la clase base.

Comportamiento
Patrón 04 / 04 · intención
State
Cada estado es una clase. El contexto delega.
Intención

Permitir que un objeto cambie su comportamiento cuando cambia su estado interno. Da la sensación de que el objeto cambió de clase.

El objeto principal (el contexto) delega en el objeto-estado que tiene puesto en ese momento.

Analogía

Un semáforo. En rojo, “avanzar” no hace nada. En verde, te deja pasar. Es el mismo objeto, pero responde distinto según su estado — y él mismo sabe a qué estado le toca ir después.

Pará y pensá — el switch de estados

Nuevo → Pagado → Enviado
class Pedido { private estado = "nuevo"; pagar() { switch (this.estado) { case "nuevo": this.estado = "pagado"; break; case "pagado": console.log("ya pagado"); break; case "enviado": console.log("ya enviado"); break; } } enviar() { switch (this.estado) { // el MISMO switch, otra vez case "nuevo": console.log("no se puede sin pagar"); break; case "pagado": this.estado = "enviado"; break; case "enviado": console.log("ya enviado"); break; } } cancelar() { /* ... y otro switch idéntico */ } }
  • Entra un estado nuevo (“en preparación”). ¿Cuántos switch tocás? ¿Y si te olvidás de uno? ¿El compilador te avisa?
  • La regla “de Pagado solo se pasa a Enviado”, ¿está en un lugar o desparramada por todos los switch?
  • Si cada estado fuera una clase, ¿qué métodos tendría? ¿Y quién decidiría el siguiente estado?

State — señal delatora

sospechalo cuando…
  • El objeto tiene un ciclo de vida con etapas y lo que se puede hacer depende de la etapa: pedido, ticket, reproductor (play/pausa/stop), documento (borrador/revisión/publicado), cajero.
  • Ves el mismo switch (estado) repetido en varios métodos.
  • Hay transiciones con reglas: “de acá solo se puede pasar a…”.

State — el código

el contexto no tiene ni un if
// Cada estado sabe DOS cosas: qué se puede // hacer, y a qué estado se pasa después. interface EstadoPedido { nombre(): string; pagar(p: Pedido): void; enviar(p: Pedido): void; } // El CONTEXTO: no tiene ni un solo if. // Delega todo en su estado actual. class Pedido { private estado: EstadoPedido = new Nuevo(); cambiarEstado(e: EstadoPedido): void { this.estado = e; } pagar(): void { this.estado.pagar(this); } // delega enviar(): void { this.estado.enviar(this); } // delega estadoActual(): string { return this.estado.nombre(); } } const p = new Pedido(); p.enviar(); // No se puede enviar sin pagar. p.pagar(); // Pago recibido. → ahora es Pagado p.enviar(); // Pedido despachado. → ahora Enviado
class Nuevo implements EstadoPedido { nombre() { return "Nuevo"; } pagar(p: Pedido) { console.log("Pago recibido."); p.cambiarEstado(new Pagado()); // ← transición } enviar(p: Pedido) { console.log("No se puede sin pagar."); } } class Pagado implements EstadoPedido { nombre() { return "Pagado"; } pagar(p: Pedido) { console.log("Ya está pagado."); } enviar(p: Pedido) { console.log("Pedido despachado."); p.cambiarEstado(new Enviado()); } } class Enviado implements EstadoPedido { nombre() { return "Enviado"; } pagar(p: Pedido) { console.log("Ya está pagado."); } enviar(p: Pedido) { console.log("Ya fue enviado."); } }

Estado nuevo (EnPreparacion) = una clase nueva. Pedido no se toca. Y la regla “de Pagado se pasa a Enviado” está en un solo lugar: adentro de Pagado.

→ Template Method. El esqueleto es fijo y solo cambian algunos pasos → herencia con un template method.

Esqueleto fijo + rellenos → Template Method; algoritmo entero intercambiable en caliente → Strategy.

Repaso relámpago — creacionales y estructurales

quiz oral: decilo vos primero

Te digo el nombre y vos tirás intención + señal delatora, sin mirar.

Singleton · Factory Method · Abstract Factory · Builder · Adapter · Composite · Decorator · Facade
Click para revelar
  • Singleton — una única instancia global · “una sola en todo el sistema”.
  • Factory Method — decidir qué clase concreta crear · “según el tipo, crear un X distinto”.
  • Abstract Factory — una familia coherente · “variante completa que no se mezcla”.
  • Builder — armar paso a paso con muchas partes · “muchos opcionales / constructor telescópico”.
  • Adapter — traducir entre interfaces incompatibles · “no encaja y no lo puedo tocar”.
  • Composite — árbol todo-parte · “contiene otros del mismo tipo, a cualquier profundidad”.
  • Decorator — sumar comportamientos combinables · “opcional, combinable, en runtime, sin tocar la clase”.
  • Facade — esconder un subsistema · “un punto de entrada único, esconder la complejidad”.

Repaso relámpago — comportamiento

la familia que más entró hoy

Ídem: intención en una línea + señal delatora.

Strategy · Observer · Chain of Responsibility · Template Method · State
Click para revelar
  • Strategy — elegir cómo hacer algo entre algoritmos · “distintas formas de calcular/pagar, elegible en runtime”.
  • Observer — un cambio avisa a varios suscriptos · “cuando pasa X, notificar a los interesados”.
  • Chain of Responsibility — la petición recorre una cadena y alguien corta · “escala / si no puede, que lo pase al siguiente”.
  • Template Method — esqueleto fijo, la subclase rellena pasos · “el procedimiento es el mismo, cambia el cómo”.
  • State — el comportamiento depende de la etapa · “según el estado / no se puede X hasta Y”.

Ejercicio en vivo — Caso 1: el checkout

el parecido a descartar: Adapter
El caso

Para procesar una venta, el checkout llama, en orden exacto, a Stock.reservar(), Precios.calcular(), Pagos.cobrar(), Facturacion.emitir(), Mail.enviar() y Stock.confirmar().

Ese bloque de seis llamadas está copiado en la web, la app móvil y el panel de admin. Si cambia el orden o se suma un paso, hay que tocar los tres lugares.

Antes de contestar

¿Quién te obliga a la interfaz que ofrecés: alguien, o la inventás vos? ¿A cuántas clases envolvés?

Click para revelar

→ Facade. Estructurar. Seis clases, un orden exacto, bloque duplicado en tres clientes. Una CheckoutFacade.procesarVenta(carrito) orquesta adentro; los tres llaman una línea.

¿Por qué no Adapter? Nadie te obliga a una interfaz existente ni hay incompatibilidad — inventás una cómoda. Facade simplifica; Adapter traduce.

Ejercicio en vivo — Caso 2: el banco

el parecido a descartar: Decorator
El caso

Una fintech evalúa créditos. La solicitud pasa por controles: identidad, deudas en el Veraz, ingreso mínimo y tope de monto.

Si cualquiera falla, se rechaza ahí mismo y los siguientes ni se ejecutan. Y cada sucursal usa controles distintos y en distinto orden: Córdoba no aplica el tope, Rosario suma uno de antigüedad al principio.

Antes de contestar

¿Se ejecutan todos los pasos y suman, o el primero que falla corta?

Click para revelar

→ Chain of Responsibility. Comunicar. Recorre una secuencia, cualquiera corta (“se rechaza ahí mismo y los siguientes ni se ejecutan”), y el orden se configura por sucursal.

¿Por qué no Decorator? Ahí todas las capas suman. Acá el primer control que falla corta. Decorator acumula; CoR descarta.

Ejercicio en vivo — Caso 3: el delivery

el parecido a descartar: Strategy
El caso

Un pedido pasa por En preparación → En camino → Entregado, y puede cancelarse.

Se puede cancelar mientras está en preparación, pero no una vez que salió. Se marca entregado solo si está en camino. Hoy el código tiene un estado: string y cuatro métodos, cada uno con el mismo switch de cinco casos.

Antes de contestar

¿Hay reglas sobre a qué se puede pasar después? ¿Quién elige el siguiente: el cliente o el propio objeto?

Click para revelar

→ State. Comunicar. Ciclo de vida con etapas, transiciones prohibidas (“no cancelar una vez que salió”) y el síntoma: el switch repetido en los cuatro métodos.

¿Por qué no Strategy? El estado elige el siguiente desde adentro, hay reglas de transición, y contesta “¿qué puedo hacer ahora?”, no “¿cómo hago X?”.

Ejercicio en vivo — Caso 4: el editor de texto

el parecido a descartar: Chain of Responsibility
El caso

El editor exporta un documento. Al exportar, el usuario marca con checkboxes si va numerado, con marca de agua, con índice automático, o cualquier combinación.

La clase Documento ya existe, funciona y no se toca. La combinación la elige el usuario en el momento, no está definida de antemano.

Antes de contestar

Si marca las tres, ¿se aplican las tres, o una sola “atiende” y corta?

Click para revelar

→ Decorator. Estructurar. Funcionalidades opcionales y combinables (2³ = 8), en runtime, sobre una clase que no se toca.

¿Por qué no CoR? Es el Caso 2 dado vuelta: acá todas las capas suman y nadie corta. Decorator acumula, CoR descarta. Si explicás por qué el 2 es CoR y el 4 es Decorator, ese par ya no te lo saca nadie.

Después de clase — Ejercicios (A a D)

familia · patrón · por qué no el parecido · UML + esqueleto TS
  1. El chat de soporte. Primero atiende un bot; si no puede, pasa a un agente nivel 1; si no, escala a nivel 2; y si tampoco, a un supervisor. Cada nivel decide si resuelve o escala. El cliente no sabe quién va a atenderlo.
  2. Los reportes. PDF, Excel y HTML. El procedimiento es idéntico: traer datos, encabezado, filas, pie con fecha, guardar. Lo único que cambia es cómo se arma el encabezado, cómo se vuelca cada fila y con qué extensión. Hoy hay tres clases con el mismo código copiado, ya desincronizadas.
  1. El reproductor. play, pausa, stop. Lo que hace cada botón depende de la situación: detenido → play arranca; reproduciendo → play no hace nada y pausa suspende; pausado → play retoma. Desde detenido no se puede pasar a pausado. Hoy hay un switch (situacion) en cada método.
  2. El backup. Tu módulo llama en orden a Compresor.zip(), Cifrador.encriptar(), Nube.subir() y Registro.log(). Ese bloque está pegado en tres comandos. Querés que cada uno llame a hacerBackup(datos) y nada más, y la interfaz hacerBackup la definís vos.

Soluciones — Ejercicios A a D

intentalos antes de leer
A → Chain of Resp.

Comunicar. “escala” + “si no puede, que lo pase al siguiente”. Bot, Nivel1, Nivel2, Supervisor; cada uno resuelve o llama a super.atender().

No es Decorator: acá uno solo atiende y la cadena se corta.

B → Template Method

Comportamiento. “el procedimiento es idéntico” + “lo único que cambia es cómo”. Reporte con generar() (template) y armarEncabezado(), volcarFila(), extension() abstractos.

No es Strategy: se rellenan pasos de un esqueleto fijo, con herencia, no se cambia el algoritmo entero.

C → State

Comportamiento. Ciclo de vida (detenido/reproduciendo/pausado), transiciones prohibidas (“desde detenido no a pausado”) y el switch repetido. Detenido, Reproduciendo, Pausado; el reproductor delega.

No es Strategy: cada estado decide el siguiente; el reproductor no lo elige desde afuera.

D → Facade

Estructurar. Cuatro clases, orden fijo, bloque duplicado en tres comandos, y la interfaz hacerBackup la inventás vos. BackupFacade orquesta adentro.

No es Adapter: no hay interfaz preexistente ni incompatibilidad. Facade simplifica; Adapter traduce.

Simulacro de parcial — el sistema de tickets

un solo enunciado, cuatro patrones

Diseñás el módulo de soporte de una empresa. Cuatro requerimientos. Identificá cada patrón, justificá y descartá el parecido — y explicá cómo se encadenan.

  1. Configuración (mail admin, límites, plantillas): se carga una sola vez y es la misma para toda la app; cualquier módulo la consulta.
  2. Crear un ticket: título y descripción obligatorios; opcionales prioridad, categoría, adjuntos, etiquetas, SLA, cliente. Hoy un new Ticket(...) de 8 parámetros.
  3. Filtros de admisión: spam, cliente activo, duplicados. Si cualquiera lo marca, se descarta y los siguientes no corren. Cada cola usa filtros y orden distintos.
  4. Ciclo de vida: Abierto → En progreso → Resuelto → Cerrado. Desde Cerrado no se vuelve a Abierto; solo se resuelve si está En progreso. Hoy el mismo switch en cada método.
Click para revelar
  • (1) → Singleton. “una sola vez / la misma para toda la app”. Ojo: viola SRP y DIP (mejor inyectarla).
  • (2) → Builder. Constructor telescópico (2 + 6 opcionales). No es Abstract Factory: un solo producto con partes.
  • (3) → Chain of Responsibility. Filtros que cortan, orden por cola. No es Decorator (ahí todos suman).
  • (4) → State. Transiciones prohibidas + switch repetido. No es Strategy (el estado decide el siguiente).

Los dos de comunicar conviven: CoR filtra la entrada (una pasada, alguien corta), State gobierna el ciclo (etapas con transiciones).

Batería — respuestas

ojo con las trampas plantadas
  1. Adapter — envolver algo incompatible que no controlás y traducir.
  2. Composite — árbol todo-parte (ítems y submenús).
  3. Singleton — “una sola” instancia.
  4. Strategy — elegir cómo calcular, sin transiciones.
  1. Observer — un cambio avisa a varios.
  2. Decorator — opcionales y combinables, se acumulan.
  3. State — ciclo de vida con transiciones.
  4. Abstract Factory — una familia coherente, sin mezclar.

Trampas: el 1 tienta con Facade (pero hay incompatibilidad + una clase → traduce); el 6 tienta con CoR (pero acumula); el 7 tienta con Strategy (pero hay transiciones). Si caíste, volvé a la ficha y a la tabla de desempates.
