El método, otra vez (no lo sueltes)

¿de qué se queja el problema?
Creacional

“No sé qué instanciar / el new está desparramado / quiero controlar la creación.”

crear · instanciar · única instancia · familia de productos · paso a paso

Estructural

“Tengo que hacer encajar / envolver / esconder / representar un todo-y-sus-partes.”

adaptar · envolver · árbol · jerarquía · esconder complejidad

Comportamiento

“Tengo que elegir un comportamiento / avisar / coordinar quién hace qué.”

El hilo de siempre: casi todos los patrones se resuelven con las mismas dos herramientas — programar contra una interfaz y composición. Si te perdés, buscá esas dos y el patrón aparece solo.

Creacional
Patrón 01 / 04 · intención
Singleton
Molde: intención → señal → código → UML → SOLID
Intención

Garantizar que una clase tenga una única instancia en todo el sistema y proveer un punto de acceso global a ella.

Hace dos cosas, no una: (1) impide que se cree más de un objeto, y (2) ofrece la manera de conseguir ese objeto desde cualquier lado.

Analogía

El gobierno de un país. Puede cambiar quién lo integra, pero “el gobierno de Argentina” es uno solo y todos saben cómo llegar a él. No podés fundar un segundo gobierno paralelo porque te haga falta.

Singleton — señal delatora

sospechalo cuando…
  • Necesitás que exista exactamente un objeto de algo: la configuración, un pool de conexiones, un logger, la caché.
  • Crear una segunda instancia sería un error o un desperdicio (dos conexiones a la base donde debería haber una).
  • El enunciado dice: “una única instancia”, “accesible desde todo el sistema”, “cargada una sola vez en memoria”.

Razoná — ¿y por qué no una variable global?

pensalo antes de pasar

Si lo único que quiero es “un objeto único al que todos accedan”… una variable global hace eso. O una clase con todos los métodos static.

  • ¿Qué le podés hacer a una clase Singleton que a una clase estática no? (pista: pensá en extends y en implements)
  • Si la configuración tarda 2 segundos en cargar de disco, ¿cuándo querés que se cargue: al arrancar o la primera vez que alguien la pide? ¿Cuál de las dos formas te deja elegir?
Click para revelar
La respuesta de la cátedra

El Singleton “es más plástico que una clase del tipo final con funciones estáticas, ya que puede ser extendida y sus métodos sobrescritos”.

O sea: es un objeto. Puede implementar interfaces, heredarse, y crearse en el momento en que se lo necesita (lazy).

Una variable global es un dato colgado. Un Singleton es un objeto con reglas.

Singleton — el código

constructor privado · atributo estático · getInstancia()
configuracion.ts
class Configuracion { // 1. la única instancia, guardada // en la propia clase private static instancia: Configuracion | null = null; private valores = new Map<string, string>(); // 2. constructor PRIVADO: nadie de // afuera puede hacer new private constructor() { this.valores.set("moneda", "ARS"); }
// 3. único punto de acceso static getInstancia(): Configuracion { if (Configuracion.instancia === null) { Configuracion.instancia = new Configuracion(); // ← lazy } return Configuracion.instancia; } get(clave: string) { return this.valores.get(clave); } set(clave: string, valor: string) { this.valores.set(clave, valor); } } const a = Configuracion.getInstancia(); const b = Configuracion.getInstancia(); console.log(a === b); // true → el MISMO objeto a.set("moneda", "USD"); console.log(b.get("moneda")); // "USD" → lo que cambia uno, lo ve el otro

    guarda la instancia
  • Constructor privado
    bloquea el new desde afuera

Inicialización perezosa: el objeto se crea recién en el primer pedido, no al arrancar el programa.

Pará y pensá — la dependencia que no se ve

la misma clase, dos versiones
// Versión A — se consigue la config por adentro class ServicioDePagos { cobrar(monto: number) { const moneda = Configuracion.getInstancia().get("moneda"); // ... } } // Versión B — la config se la pasan por constructor class ServicioDePagos { constructor(private config: Configuracion) {} // ← cobrar(monto: number) { const moneda = this.config.get("moneda"); // ... } }
  • Imaginate que abrís el archivo y solo leés la primera línea de cada clase. En la B, ¿de qué depende ServicioDePagos? ¿Y en la A?
  • En la A la dependencia existe igual (el acoplamiento está), pero no aparece en ningún lado visible. ¿A qué principio de la Clase 2 le está faltando el respeto?
  • Contá cuántas responsabilidades tiene la clase Configuracion. No una: mirá qué hace además de guardar valores.

B pide lo que necesita. A se lo agarra. Y lo que una clase se agarra sola, nadie se lo puede cambiar.

Singleton — la letra chica (cuándo NO usarlo)

el patrón más criticado del catálogo
Ventajas
  • Certeza de que hay una sola instancia, y un punto de acceso claro.
  • Inicialización perezosa: se crea recién cuando alguien lo pide.
Desventajas — y son serias
  • Viola SRP: la clase hace su trabajo y además controla su propia instanciación. Dos motivos de cambio.
  • Viola DIP y esconde las dependencias: quien lo usa lo consigue por adentro, así que la dependencia no aparece en la firma. El acoplamiento existe, pero invisible.
  • Es estado global disfrazado: cualquiera puede cambiarle un valor y todos lo ven. “¿Quién me cambió la moneda a USD?” → puede ser cualquiera de las 200 clases.
  • Multihilo: dos hilos pueden entrar juntos al if y crear dos instancias. En TS/JS no te pasa, pero es el clásico bug en Java o C#.

Regla práctica: antes de meter un Singleton, preguntate si no alcanza con pasar el objeto por constructor. Casi siempre alcanza. Si el enunciado no dice explícitamente “una única instancia”, probablemente no sea Singleton.

Creacional
Patrón 02 / 04 · intención
Abstract
Factory
La palabra clave es familia.
Intención

Crear familias de objetos relacionados sin especificar sus clases concretas, y garantizar que los productos de una misma familia se usen juntos.

No es “un producto”: son varios productos que tienen que ser coherentes entre sí.

Analogía

Comprar muebles de un mismo juego. Si elegís “moderno”, querés la silla moderna, la mesa moderna y el sillón moderno. Lo que no querés es una silla moderna con una mesa victoriana.

Abstract Factory — señal delatora

sospechalo cuando…
  • Tenés varios productos que varían juntos por “estilo”, “plataforma”, “proveedor” o “tema”: botón + checkbox + menú; conexión + comando + lector de una base.
  • El enunciado te pide que no se mezclen: “si el sistema corre en Windows, toda la interfaz debe ser de Windows”.
  • Palabras que lo delatan: familia de productos, conjunto coherente, variante completa, skin / tema / plataforma / proveedor.
productos abstractos
// el cliente solo conoce // estas interfaces interface Boton { render(): string; } interface CheckBox { render(): string; }

Dos tipos de producto distintos. Ese “dos” es toda la diferencia con Factory Method.

Abstract Factory — el código

la coherencia queda garantizada por construcción

Formulario no puede producir un botón Mac con un checkbox de Windows aunque se lo proponga. La coherencia no depende de que el programador se acuerde: está garantizada por construcción.

Razoná — Factory Method vs Abstract Factory

este par cae seguro en el parcial
Enunciado 1

“Según el nivel del juego, la lógica de oleadas tiene que crear un enemigo distinto: lobo, murciélago o caballero.”

Enunciado 2

“El juego tiene modo Bosque y modo Cueva. En cada modo cambian el enemigo, el sonido ambiente Y la textura del piso, y nunca se pueden mezclar: no puede aparecer un lobo con el sonido de la cueva.”

  • ¿Cuántos tipos distintos de producto hay en cada enunciado? Contalos.
  • ¿En cuál el problema es “¿qué clase concreta instancio?” y en cuál es “¿cómo garantizo que todo lo que creo sea del mismo juego?”
Click para revelar

Enunciado 1 → Factory Method. UN producto, varias variantes; una subclase decide cuál.

Enunciado 2 → Abstract Factory. VARIOS productos que varían en bloque y no pueden mezclarse.

La regla corta: Factory Method = un producto. Abstract Factory = una familia.

De hecho, una Abstract Factory está hecha por dentro de varios Factory Methods: crearBoton() y crearCheckBox() son, cada uno, un método fábrica.

Creacional
Patrón 03 / 04 · intención
Builder
Cuando el new se vuelve un monstruo.
Intención

Separar la construcción de un objeto complejo de su representación, para armarlo paso a paso y que el mismo proceso produzca objetos distintos.

Se usa cuando el objeto tiene muchas partes opcionales.

Analogía

Pedir una pizza. No hay un único “constructor de pizza” con 12 parámetros: elegís la masa, después la salsa, después vas agregando ingredientes. Paso a paso, y parás donde quieras.

Pará y pensá — el constructor telescópico

solo con esta línea, ¿qué pizza pidió?
const p = new Pizza("a la piedra", "tomate", ["muzzarella"], "grande", true, false, false);

La masa, la salsa, los ingredientes y el tamaño los leés sin problema. Pero el true y los dos false no te dicen nada: para saber qué son tenés que ir a abrir la clase. Ese viaje, en cada línea que leas, es el primer costo.

ahora sí, la definición
constructor(masa, salsa, ingredientes, tamanio, bordeRelleno: boolean, extraQueso: boolean, sinTACC: boolean) {}
  • Si me equivoco y escribo false, true, false, el borde deja de estar relleno y aparece extra queso. ¿El compilador me avisa? Los tres son boolean…
  • Si querés una pizza solo con masa integral, ¿qué escribís en los otros 6 lugares? ¿Podés saltear los que no te importan?
  • Mañana agregan “sin sal”. ¿Qué les pasa a todas las llamadas a new Pizza(...) que ya existen?
Click para revelar
Constructor telescópico

Cada opción nueva le suma un parámetro. El orden se vuelve imposible de recordar, el compilador no te protege (todos son del mismo tipo) y no podés omitir lo que no usás.

Guardá esa incomodidad: ese es el olor que cura Builder.

Builder — señal delatora

sospechalo cuando…
  • El objeto tiene muchos atributos opcionales y verías un constructor con un montón de parámetros (o cinco constructores sobrecargados).
  • La construcción tiene pasos y querés controlarlos: “armar el objeto paso a paso”.
  • Querés el mismo proceso de armado produciendo representaciones distintas (una napolitana y una fugazzeta salen del mismo builder).

cada valor viaja con su nombre pegado
class Pizza { masa = "clásica"; salsa = "tomate"; tamanio = "mediana"; bordeRelleno = false; ingredientes: string[] = []; descripcion(): string { const borde = this.bordeRelleno ? ", con borde relleno" : ""; return `Pizza ${this.tamanio} de masa ${this.masa}${borde}` + ` — ${this.ingredientes.join(", ")}`; } } // Y ahora lo uso: un método por vez. const b = new PizzaBuilder(); b.tamanio("grande"); b.conMasa("a la piedra"); b.conBordeRelleno(); b.agregar("muzzarella"); const p = b.construir(); console.log(p.descripcion()); // Pizza grande de masa a la piedra, // con borde relleno — muzzarella
class PizzaBuilder { private pizza = new Pizza(); // la pizza a medio armar // Cada paso configura UNA cosa y devuelve el propio // builder (return this). conMasa(m: string): PizzaBuilder { this.pizza.masa = m; return this; } conSalsa(s: string): PizzaBuilder { this.pizza.salsa = s; return this; } tamanio(t: string): PizzaBuilder { this.pizza.tamanio = t; return this; } conBordeRelleno(): PizzaBuilder { this.pizza.bordeRelleno = true; return this; } agregar(i: string): PizzaBuilder { this.pizza.ingredientes.push(i); return this; } // el ÚNICO que no devuelve el builder: // devuelve la pizza ya terminada construir(): Pizza { return this.pizza; } }

Compará con new Pizza("a la piedra", "tomate", [...], "grande", true, false, false). Acá el orden no importa, conBordeRelleno() no se confunde con nada, y lo que no te interesa no lo escribís.

Builder — por qué cada paso hace return this

las dos formas son lo mismo

¿Para qué devuelve el builder cada método, si el objeto b ya lo tenía? Para poder encadenar. Como b.tamanio("grande") devuelve el mismo b, podés escribirle un punto y seguir llamando métodos sobre el resultado.

sin encadenar
const b = new PizzaBuilder(); b.tamanio("grande"); b.conMasa("a la piedra"); b.conBordeRelleno(); b.agregar("muzzarella"); const p = b.construir();
encadenado — es lo mismo
const p = new PizzaBuilder() .tamanio("grande") // devuelve el builder... .conMasa("a la piedra") // ...le pido otra cosa .conBordeRelleno() // ...y otra .agregar("muzzarella") // ...y otra .construir(); // ← corta la cadena: da la Pizza

Esto se llama interfaz fluida (fluent interface). Ojo: es una comodidad de lectura, no es el patrón. Un Builder cuyos métodos devuelvan void sigue siendo un Builder — lo que lo define es construir el objeto por partes y entregarlo terminado al final.

Builder — el Director (opcional)

recetas fijas que se repiten
el Director conoce las recetas
class Pizzeria { // ← el Director napolitana(b: PizzaBuilder): Pizza { return b.tamanio("grande") .conMasa("clásica") .agregar("muzzarella") .agregar("tomate") .agregar("ajo") .construir(); } } const napo = new Pizzeria() .napolitana(new PizzaBuilder());

El Director sabe en qué orden llamar a los pasos.

El Builder sabe cómo se hace cada paso.

Sirve cuando el mismo armado se repite en muchos lados. Si tu receta es única y no se repite, el Director es opcional: no lo metas por meterlo.

Trade-off (cátedra): Builder aumenta la cantidad de clases del sistema. Si tu objeto tiene 3 atributos y ninguno opcional, no uses Builder: usá el constructor y listo.

Checkpoint — los 4 creacionales

decilo vos: ¿cuál, y por qué no el parecido?
  1. “Necesitamos una única conexión al servidor de la cátedra, compartida por todo el sistema.”
  2. “Según el tipo de archivo, hay que crear un objeto Documento distinto; el flujo de apertura es siempre igual.”
  3. “Un usuario se registra con nombre y mail obligatorios, y opcionalmente foto, bio, teléfono, fecha de nacimiento y preferencias de notificación.”
  4. “La app tiene tema claro y tema oscuro. Cada tema define su propio botón, panel e ícono, y no se pueden mezclar.”

  • 1) Singleton — una única instancia.
  • 2) Factory Method — UN producto, varias variantes concretas.
  • 3) Builder — muchos atributos opcionales → constructor telescópico.
  • 4) Abstract Factory — una familia de productos que varían en bloque.
El desempate creacional

¿uno solo? → Singleton · ¿un producto, varias variantes? → Factory Method · ¿varios productos coherentes? → Abstract Factory · ¿un producto con partes opcionales? → Builder

Estructural
Patrón 04 / 04 · intención
Composite
El cliente no pregunta si es hoja o rama.
Intención

Componer objetos en estructuras de árbol para representar jerarquías todo-parte, y lograr que el cliente trate de la misma manera a un objeto individual (hoja) y a un grupo (rama).

Analogía

Una caja de mudanza. Adentro puede haber cosas sueltas o más cajas, que adentro tienen más cosas o más cajas. “¿Cuánto pesa esa caja?” es la misma operación sin importar la profundidad: sumar lo que hay adentro.

Razoná — el árbol y el if

la solución “a mano”
function tamanio(nodo: any): number { if (nodo instanceof Archivo) return nodo.kb; if (nodo instanceof Carpeta) { let total = 0; for (const hijo of nodo.hijos) total += tamanio(hijo); // ← ojo acá return total; } return 0; }
  • Si mañana aparece un tercer tipo de nodo (un acceso directo), ¿cuántos if como este hay en el sistema? Uno para el tamaño, otro para listar, otro para buscar…
  • Ese instanceof, ¿qué principio de la Clase 2 te está gritando que estás violando?
  • La importante: ¿y si tamanio() fuera un método que tienen tanto el archivo como la carpeta? ¿Qué haría la versión de la carpeta?
Click para revelar

La cura: que hoja y rama compartan la interfaz. La rama implementa la operación delegando en sus hijos — y como los hijos también son de la misma interfaz, la recursión sale sola. El if desaparece: el polimorfismo lo reemplaza.

Composite — señal delatora

sospechalo cuando…
  • Aparece una estructura de árbol o jerárquica: menú con submenús, carpetas con carpetas, comentarios con respuestas, organigrama, categorías con subcategorías.
  • El enunciado usa: “contiene”, “agrupa”, “subgrupo”, “a cualquier nivel de profundidad”, “un ítem o un conjunto de ítems”.
  • Querés hacer la misma operación sobre uno o sobre muchos sin distinguirlos.
El detalle fino de la cátedra

Composite tiene sentido cuando el comportamiento de la rama y el de la hoja son distintos (la rama tiene que recorrer a sus hijos).

Si fueran iguales, alcanzaría con una clase que tenga una lista adentro — no hace falta el patrón.

La magia está en una sola cosa: el cliente no pregunta si es hoja o rama. Le habla igual a los dos.

Composite — el código

la recursión sale sola, sin un solo if
// Component: la interfaz común. // La hoja y la rama la implementan IGUAL. interface NodoFS { nombre(): string; tamanio(): number; } // Leaf (hoja): no tiene hijos. Sabe su tamaño y listo. class Archivo implements NodoFS { constructor(private _nombre: string, private kb: number) {} nombre(): string { return this._nombre; } tamanio(): number { return this.kb; } } // El cliente NO pregunta si es archivo o carpeta. // Le habla igual a los dos: const cosas: NodoFS[] = [raiz, new Archivo("suelto.txt", 1)]; cosas.forEach(c => console.log(c.nombre(), c.tamanio())); // proyecto 23 ← 12 + 8 + 3, recursivo // suelto.txt 1
// Composite (rama): tiene hijos, y resuelve la // operación DELEGANDO en ellos. class Carpeta implements NodoFS { private hijos: NodoFS[] = []; constructor(private _nombre: string) {} agregar(n: NodoFS): void { this.hijos.push(n); } quitar(n: NodoFS): void { this.hijos = this.hijos.filter(h => h !== n); } nombre(): string { return this._nombre; } tamanio(): number { // mismo método que la hoja, pero se apoya // en los hijos → recursión natural return this.hijos.reduce( (total, hijo) => total + hijo.tamanio(), 0); } } const raiz = new Carpeta("proyecto"); const src = new Carpeta("src"); src.agregar(new Archivo("main.ts", 12)); src.agregar(new Archivo("utils.ts", 8)); raiz.agregar(src); raiz.agregar(new Archivo("README.md", 3));

Carpeta.tamanio() llama a tamanio() de sus hijos. Como un hijo puede ser otra carpeta, la recursión baja sola hasta las hojas. Cero instanceof.

Composite — el trade-off que marca la cátedra

no hay salida gratis

La pregunta de diseño: ¿dónde ponés agregar() y quitar()? Las dos opciones tienen costo, y en el parcial lo que se evalúa es que sepas explicar el intercambio, no que memorices una versión.

Opción A — en la interfaz común

Ponés agregar() y quitar() en NodoFS, así el cliente trata exactamente igual a la hoja y a la rama. Transparencia total.

El costo: Archivo queda obligado a implementar métodos que no tienen sentido para él. ¿Qué le agregás a un archivo?

Viola ISP
Opción B — solo en la rama

Dejás agregar() y quitar() solo en Carpeta (es lo que hicimos en el código). Cada clase implementa únicamente lo que le sirve.

El costo: si el cliente quiere agregar un hijo, tiene que preguntar el tipo — vuelve el instanceof que queríamos matar.

Salva ISP · rompe la uniformidad

Conexión SOLID del patrón: OCP (sumás un tipo de nodo sin tocar al cliente) y LSP (hoja y rama son sustituibles entre sí)… pero a costa de tensionar ISP. Composite es el ejemplo clásico de que los principios a veces compiten entre sí.

Ejercicio en vivo — La inmobiliaria

la pista solo si te trabás
El caso

El portal se despliega para dos marcas: Premium y Económica.

En Premium, la ficha de propiedad se muestra con galería de fotos grande, el mapa es interactivo y el contacto es por teléfono directo.

En Económica, la ficha es una lista simple, el mapa es una imagen estática y el contacto es un formulario.

La regla del negocio: nunca puede aparecer una ficha Premium con un mapa estático. Cada marca es un paquete completo o nada.

Antes de contestar
  • Contá los tipos de producto que aparecen en el enunciado.
  • Preguntate si el problema es “¿qué clase concreta instancio?” o “¿cómo garantizo que todo lo que creo sea del mismo juego?”.
Pista — click

El parecido a descartar: Factory Method. Los dos evitan el new cableado y fabrican objetos…

La solución

lo que vale es el “por qué no el parecido”
Crear
→ Abstract
Factory

Hay tres productos (ficha, mapa, contacto) que varían en bloque según la marca, y el enunciado exige coherencia: “nunca una ficha Premium con un mapa estático”, “un paquete completo o nada”.

Esa imposibilidad de mezclar es la firma del patrón.

¿Por qué no Factory Method?

Factory Method fabrica UN producto con variantes. Acá son varios productos distintos que tienen que ser consistentes entre sí.

Contá los tipos de producto: si es uno → Factory Method. Si son varios que viajan juntos → Abstract Factory.

De hecho, una Abstract Factory está hecha por dentro de varios Factory Methods: crearFicha(), crearMapa(), crearContacto().

Factory Method vs Abstract Factory → un producto vs. una familia.

Abstract Factory vs Builder → varios productos coherentes entre sí vs. uno con muchas partes opcionales.

Después de clase — Ejercicios (1 a 6)

dos de los seis son de la Clase 3

Para cada uno: 1) ¿el problema es de crear, estructurar o comunicar? · 2) ¿qué patrón, y por qué no el parecido? · 3) bosquejá el UML y un esqueleto en TypeScript.

  1. El logger. Si dos partes del programa crean cada una su propio logger, el archivo se corrompe. Necesitás que exista un solo logger en toda la app y que cualquier módulo pueda conseguirlo sin recibirlo por parámetro desde diez niveles arriba.
  2. Retail y Mayorista. En Retail el precio muestra IVA incluido, el carrito permite hasta 10 unidades y el mail es informal. En Mayorista el precio va sin IVA, sin tope, y el mail es formal con datos fiscales. Nunca puede salir un mail informal con precios sin IVA: coherente o nada.
  3. Alta de propiedad. Dirección y precio obligatorios; opcionales: metros cubiertos, totales, ambientes, cocheras, mascotas, amenities, fotos y descripción. Hoy hay un constructor con 10 parámetros y nadie se acuerda del orden.
  1. La consultora. Un equipo puede tener empleados y otros equipos adentro, sin límite de profundidad. Hay que calcular el costo salarial total de un empleado suelto, de un equipo, o de la empresa entera.
  2. El servicio SOAP. Tu módulo de reportes consume datos con FuenteDatos.obtener(): Registro[]. Un cliente nuevo expone su información con un servicio SOAP que tiene fetchAll() y devuelve XML con otros nombres de campo. No podés modificar el servicio del cliente ni querés reescribir el módulo.
  3. La app de notas. Exportar una nota puede ir cifrada, comprimida, con marca de agua, o cualquier combinación de las tres. El usuario elige la combinación en el momento de exportar, con checkboxes. La clase Nota ya existe y funciona; no la querés tocar.

Soluciones — Ejercicios 1 a 3

intentalos antes de leer
1 → Singleton

Crear. El enunciado dice literalmente “un solo logger en toda la aplicación” y “conseguirlo sin recibirlo por parámetro”: las dos mitades del patrón (instancia única + punto de acceso global).

Constructor privado, private static instancia, static getInstancia().

Decilo también en el parcial

Viola SRP (guarda logs y administra su propia instancia) y DIP (quien lo usa no declara que depende de él). Se justifica solo porque el enunciado exige el acceso global.

2 → Abstract Factory

Crear. Tres productos (cálculo de precio, política de carrito, generador de mail) que varían en bloque según el cliente, y el enunciado exige coherencia: “nunca un mail informal con precios sin IVA”.

No es Factory Method

Ahí habría un solo producto variando. Acá son varios que no se pueden mezclar — y esa imposibilidad de mezclar es la firma de Abstract Factory.

3 → Builder

Crear. Constructor telescópico de manual: 2 obligatorios + 8 opcionales. Se lee, y lo que no aplica no se escribe.

new PropiedadBuilder().enDireccion(...).conPrecio(...).conAmbientes(3).aceptaMascotas().construir()

No es Abstract Factory

Acá hay un solo producto (la propiedad) con muchas partes opcionales, no varios productos que tengan que ser coherentes entre sí.

Soluciones — Ejercicios 4 a 6

el 5 y el 6 son de la Clase 3
4 → Composite

Estructurar. Jerarquía todo-parte (un equipo contiene empleados y equipos), profundidad arbitraria, y la misma operación —costoSalarial()— sobre la hoja y sobre la rama.

Empleado (hoja) devuelve su sueldo; Equipo (rama) tiene miembros: Costeable[] y suma los de sus hijos → la recursión sale sola, sin un if.

No es Decorator

Decorator envuelve un objeto para sumarle comportamiento. Acá Equipo contiene muchos hijos y no le agrega nada a ninguno: los agrupa. Composite = árbol; Decorator = pila.

5 → Adapter
¡de la Clase 3!

Estructurar. Dos interfaces incompatibles (obtener() vs fetchAll(), Registro[] vs XML) y una que no controlás. class SoapAdapter implements FuenteDatos compone el servicio y traduce.

No es Decorator

Decorator mantiene la interfaz y le suma comportamiento. Acá no sumás nada: la interfaz del SOAP no encaja y tu único trabajo es traducirla. Adapter cambia la interfaz; Decorator la conserva.

6 → Decorator
¡de la Clase 3!

Estructurar. Funcionalidades opcionales y combinables (2³ = 8), elegidas en runtime, sobre una clase que no querés tocar.

new Cifrado(new Comprimido(new ExportadorBase(nota)))

No es Strategy

En Strategy hay un solo algoritmo activo por vez y el cliente lo reemplaza. Acá las tres pueden estar activas a la vez, apiladas. Strategy intercambia; Decorator acumula.
